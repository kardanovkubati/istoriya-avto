# История Авто: дизайн MVP

Дата: 2026-05-12
Статус: черновик, утвержденный в обсуждении

## 1. Цель и позиционирование

Рабочее имя продукта: **История Авто**.

Сервис предназначен для РФ-аудитории и помогает проверить историю автомобиля перед покупкой. Главная ценность: пользователь получает один сводный отчет по автомобилю, собранный из загруженных отчетов, объявлений и нормализованных фактов.

Публичное позиционирование не должно звучать как "обмен отчетами Автотеки/Авто.ру/Дрома". Продукт позиционируется как сервис проверки истории авто. Механика загрузки отчета используется как способ пополнения базы и получения баллов.

Принципы:

- Не показывать пользователям оригинальные отчеты сторонних сервисов.
- Не копировать стиль, тексты, иконки, визуальные шкалы, фирменную верстку и формулировки сторонних сервисов.
- Строить собственный отчет на базе нормализованных фактов.
- В пользовательском отчете не указывать названия Автотека, Авито, Авто.ру или Дром как источники конкретных фактов.
- В FAQ и инструкциях можно нейтрально упоминать поддерживаемые сервисы.
- Перед публичным запуском юридические документы, формулировки, фотографии и использование брендов должны быть проверены юристом.

## 2. MVP-срез

MVP идет по сбалансированному варианту.

Входит в MVP:

- умный поиск по ссылке объявления, VIN или госномеру;
- загрузка PDF-отчета или authless-ссылки на отчет;
- Авито/Автотека как первый и самый качественно поддержанный источник;
- архитектура под Авто.ру и Дром без блокировки первого релиза;
- гостевой сценарий на 7 дней;
- авторизация на выбор через Telegram, Max или номер телефона;
- баллы за полезные загруженные отчеты;
- подписки Start, Pro, Max;
- один сводный отчет по VIN;
- оценка прозрачности истории;
- история объявлений с фото, ценами и пробегами;
- PDF-выгрузка отчета;
- share-ссылки на 7 дней;
- Telegram и Max боты для входа, уведомлений и поддержки;
- desktop-first админка;
- жалобы и обращения поддержки;
- базовый антифрод загрузок;
- SEO-лендинги и FAQ;
- юридические страницы.

Не входит в MVP:

- OCR для сканов и картинок внутри PDF;
- технический аккаунт стороннего сервиса как fallback для ссылок с авторизацией;
- API и массовый экспорт;
- реклама;
- разовая продажа баллов;
- публичные комментарии и заметки пользователей;
- блог;
- автоматический мониторинг объявлений по расписанию;
- полноценная ролевая модель админки.

## 3. Пользовательские сценарии

### 3.1 Главный поиск

На главной странице есть одно умное поле: "Вставьте ссылку на объявление, VIN или госномер". Система сама определяет тип запроса.

Поддерживаемые типы:

- ссылка объявления;
- VIN;
- госномер.

Если пользователь вставляет неподдержанную ссылку, интерфейс сообщает, что пока поддерживаются ссылки Авито, Авто.ру и Дром, а также предлагает поиск по VIN/госномеру или загрузку PDF/ссылки на отчет.

### 3.2 Поиск по ссылке объявления

При поиске по ссылке объявления сервис точечно снимает snapshot объявления:

- источник и ad_id;
- исходный URL;
- марка, модель, год;
- поколение/кузов, если доступны;
- цвет, если доступен;
- регион/город;
- цена из объявления;
- пробег из объявления;
- описание с удалением персональных данных;
- фотографии;
- технические признаки для сопоставления.

Автоматическое обновление объявления по расписанию в MVP не делается. Snapshot фиксируется один раз в момент пользовательского запроса.

Если объявление удалено или недоступно:

- сервис сохраняет URL и ad_id, если его можно извлечь;
- пробует найти совпадения по базе;
- предлагает загрузить отчет;
- после авторизации может подписать пользователя на уведомление о появлении отчета.

### 3.3 Результаты и превью

Результаты показываются в превью-режиме.

Если есть точное совпадение, может быть показан один кандидат. Если есть похожие совпадения, показываются все найденные кандидаты с пагинацией по 3-5 крупных горизонтальных карточек на страницу.

Уровень совпадения, проценты или "высокое/среднее совпадение" пользователю не показываются. Ответственность за выбор кандидата лежит на пользователе. Перед открытием нужно ясно показать, что после открытия балл не возвращается, если пользователь выбрал не тот автомобиль.

Превью может показывать:

- фото;
- марку, модель, год;
- поколение/кузов;
- цвет;
- двигатель, КПП, привод;
- регион/город;
- последнюю цену из объявления;
- последний пробег из объявления;
- частично скрытый VIN;
- дату последней фиксации/обновления.

Превью не показывает:

- госномер;
- ДТП;
- ограничения;
- залоги;
- розыск;
- владельцев;
- историю пробегов;
- историю изменения цен;
- расчеты ремонта;
- такси, каршеринг, лизинг;
- юридические статусы;
- другие ценные блоки отчета.

### 3.4 Если отчета нет

Если отчета нет, показывается сообщение в духе:

> Отчета пока нет, но он может появиться позже. Загрузите свой отчет и получите балл на будущий просмотр.

После авторизации пользователь может подписаться на уведомление через Telegram или Max.

### 3.5 Разблокировка отчета

Логика доступа:

- если отчет по VIN уже открыт пользователю ранее, повторный просмотр бесплатный;
- если есть активная подписка с доступным лимитом новых отчетов, тратится подписочный лимит;
- если лимит подписки закончился или подписки нет, но есть баллы, списывается 1 балл;
- если нет подписочного лимита и баллов, показывается экран разблокировки: загрузить отчет или выбрать тариф.

Если пользователь пришел в загрузку из сценария разблокировки, после успешного начисления балла интерфейс спрашивает: "Открыть выбранный отчет?" Балл не списывается без этого подтверждения в данном сценарии.

В обычной загрузке получение баллов и открытие отчетов не связаны: пользователь загружает отчет, получает балл на баланс и сам решает, когда его потратить.

Один балл или один подписочный лимит открывает сводный отчет по VIN навсегда для этого пользователя, включая будущие обновления.

## 4. Загрузки отчетов

### 4.1 Поддерживаемые входы

MVP поддерживает:

- текстовый PDF-отчет;
- authless-ссылку на отчет.

MVP не поддерживает:

- OCR для PDF-сканов;
- ввод логинов/паролей сторонних сервисов;
- технический аккаунт стороннего сервиса для ссылок, требующих авторизацию.

Если ссылка не открывается без авторизации, пользователь получает предложение скачать PDF и загрузить его.

Перед загрузкой нужен чекбокс: пользователь подтверждает, что имеет право загрузить отчет, и соглашается с правилами сервиса и начисления баллов.

Пользователь может загрузить отчет по любому автомобилю. Отчет не обязан относиться к автомобилю, который пользователь хочет открыть.

### 4.2 Хранение оригиналов

Оригинальные PDF, HTML, снимки страниц и служебные артефакты ссылок хранятся закрыто до 6 месяцев.

Они используются только для:

- перепроверки парсинга;
- антифрода;
- разбора споров;
- обработки жалоб;
- улучшения парсеров.

Оригиналы не показываются обычным пользователям. Доступ к ним есть только в админке с аудитом.

## 5. Баллы

Баллы в пользовательском интерфейсе только целые.

Базовый принцип: актуальный полезный отчет дает 1 балл. Дата актуальности определяется датой формирования исходного отчета.

Дата загрузки к нам нужна для аудита и антифрода. Дата событий внутри отчета влияет на историю автомобиля, но не заменяет дату формирования отчета.

### 5.1 Правила начисления

- Отчет 0-90 дней: валидный полезный отчет дает 1 балл.
- Отчет 91-180 дней: если это первый отчет по VIN, дает 1 балл; если это дубль или отчет старее имеющегося, дает 0.
- Отчет старше 180 дней: принимается в базу, но без автоматического начисления; ручная проверка возможна.
- Более свежий отчет по уже известному VIN дает 1 балл только если этот пользователь еще никогда не получал балл за этот VIN.
- Один пользователь может получить балл за один VIN только один раз навсегда.
- Один пользователь может получить балл за конкретный исходный отчет только один раз навсегда.
- Один report_fingerprint может автоматически дать балл максимум 3 разным пользователям; дальше без автоначисления или ручная проверка.

Если отчет не дал балл в сценарии разблокировки, пользователь остается на экране выбора: загрузить другой отчет или выбрать тариф. Сам отчет может быть принят в базу, если он валиден.

### 5.2 Скорость и риск

Целевые времена:

- PDF: обычно 10-20 секунд;
- authless-ссылка: обычно 30-60 секунд.

Риск-модель:

- низкий риск: балл начисляется сразу после базовой проверки;
- средний риск: отчет принят, балл отложен;
- высокий риск: ручная проверка.

При последующем обнаружении мошенничества баланс может быть скорректирован, а аккаунт ограничен.

### 5.3 Антифрод загрузок

MVP включает базовый антифрод:

- извлечение текста напрямую, без OCR;
- проверка VIN в нескольких местах отчета;
- проверка даты формирования и ее положения в документе;
- PDF metadata: creator, producer, дата создания/изменения;
- признаки редактирования: разные шрифты, наложенные текстовые слои, странные bounding boxes;
- совпадение структуры с известными шаблонами;
- хэш файла и похожесть на уже загруженные отчеты;
- лимиты на массовые загрузки;
- анализ повторяющихся IP/устройств/паттернов;
- очередь ручной проверки для подозрительных отчетов.

Если дата формирования отчета не распарсилась, противоречит содержимому или выглядит измененной, отчет уходит в ручную проверку без автоматического начисления.

Частичный парсинг может дать балл только если есть VIN, дата формирования и минимальный набор ключевых блоков. Сильно частичные или странные отчеты идут в очередь проверки.

## 6. Авторизация и аккаунты

В MVP пользователь может войти на выбор:

- через Telegram;
- через Max;
- через номер телефона.

Телефон не обязателен, если пользователь выбрал Telegram или Max. Для платежа отдельно запрашивается email для чека, если он нужен платежному провайдеру.

### 6.1 Уникальность идентификаторов

- Один номер телефона может быть привязан только к одному аккаунту.
- Один Telegram account может быть привязан только к одному аккаунту.
- Один Max account может быть привязан только к одному аккаунту.
- Если пользователь входит через уже привязанный канал, он попадает в тот аккаунт, к которому канал привязан.
- Если в настройках пользователь пытается подключить Telegram, Max или телефон, который уже занят другим аккаунтом, интерфейс показывает ошибку.
- Автоматического объединения аккаунтов в MVP нет.

### 6.2 Гостевая сессия

Гостевая сессия живет 7 дней.

Гость может:

- искать отчет;
- видеть превью;
- выбирать кандидата;
- загружать отчет;
- получить временный балл.

Перед полным открытием отчета гость мягко переводится к авторизации.

После входа любым способом гостевая сессия переносится в аккаунт:

- временные баллы;
- выбранный отчет;
- запросы на появление отчета;
- загруженные отчеты;
- контекст поиска.

После авторизации баллы становятся обычными и бессрочными.

### 6.3 Настройки аккаунта

Настройки пользователя включают:

- уведомления;
- подключение Telegram, Max, телефона;
- основной канал связи;
- email для чеков;
- управление автопродлением;
- удаление аккаунта.

Удаление аккаунта удаляет или обезличивает только пользовательскую часть:

- контакты;
- сессии;
- настройки;
- балансы;
- подписку как пользовательский доступ;
- открытые отчеты как пользовательские доступы.

Загруженные отчеты, фотографии и нормализованные данные автомобилей остаются в базе. Перед удалением интерфейс предупреждает, что пользовательские данные, доступы и баланс будут утеряны навсегда и не восстановятся даже при повторной регистрации на тот же контакт.

## 7. Telegram и Max

Telegram и Max используются:

- для входа;
- для привязки аккаунта;
- для уведомлений;
- для поддержки.

Пользователь может подключить один или оба канала и выбрать основной.

Боты отправляют:

- отчет появился;
- проверка загрузки завершена;
- ответ поддержки;
- подписка скоро закончится;
- важные статусы по жалобам или спорным загрузкам.

Пользователь может написать вопрос в поддержку через форму или бота. Обращение создает тикет в админке. Ответ из админки приходит пользователю обратно через Telegram или Max.

## 8. Подписки и платежи

Монетизация MVP:

- подписки;
- бесплатный путь через загрузку отчетов и получение баллов.

На старте нет:

- рекламы;
- разовой продажи баллов.

Черновые тарифы:

- Start: 399 ₽ / 50 новых отчетов за биллинговый период.
- Pro: 999 ₽ / 200 новых отчетов за биллинговый период.
- Max: 1990 ₽ / 500 новых отчетов за биллинговый период.

Под новым отчетом понимается первый доступ пользователя к сводному отчету по конкретному VIN. После того как VIN открыт, отчет можно смотреть неограниченное количество раз, включая будущие обновления, без списаний лимита и баллов.

Правила подписки:

- лимит живет по личному биллинговому периоду пользователя, не по календарному месяцу;
- автопродление включено по умолчанию;
- пользователь может выключить автопродление в настройках;
- при выключении автопродления тариф работает до конца оплаченного периода;
- неиспользованные новые отчеты по подписке сгорают в конце периода;
- баллы от загрузок не сгорают;
- загрузки дают баллы всегда по правилам начисления, даже при активной подписке.

Если у пользователя есть подписка и баллы, при открытии нового отчета сначала тратится подписочный лимит. Если лимит подписки закончился, баллы списываются автоматически при нажатии "Открыть отчет".

В UI всегда должен быть виден статус пользователя:

- тариф;
- остаток новых отчетов за период;
- баланс баллов.

Платежи реализуются через абстрактный payment provider, с ориентацией на ЮKassa или CloudPayments. Для чека достаточно email.

## 9. Share и PDF

Владелец открытого отчета может:

- скачать PDF в нашем оформлении;
- создать share-ссылку;
- поделиться через Telegram, WhatsApp или ссылкой.

Share-ссылка:

- живет 7 дней;
- открывается без авторизации по токену;
- не индексируется;
- не тратит лимиты или баллы владельца;
- не дает получателю возможность скачать PDF;
- не дает получателю возможность создать новую share-ссылку.

Просмотры по share-ссылке пишутся в статистику и антиабуз-аудит.

PDF и share-версия должны показывать дату формирования нашего отчета и дисклеймер. Можно добавить мягкий watermark: ID отчета, дата или нейтральный знак сервиса.

## 10. Сводный отчет

Пользователь видит один наш сводный отчет по автомобилю, а не список исходных отчетов.

Центр отчета: VIN.

Все загрузки, объявления, фотографии и факты по одному VIN складываются внутрь, актуализируют историю и показываются как единая карточка.

В пользовательском отчете показывается нейтральная строка:

> История составлена на основе N загруженных отчетов. Последнее обновление: дата.

Для чувствительных блоков показывается свежесть данных. Если последний отчет старше 90 дней, рядом с критичными выводами должно быть ясно указано, что данные могли измениться после даты отчета.

### 10.1 Разделы отчета

Порядок разделов:

1. Сводка и оценка прозрачности истории.
2. Паспорт авто и идентификаторы.
3. Юридические риски: ограничения, залог, розыск.
4. ДТП и расчеты ремонта.
5. Пробеги и расхождения.
6. История владельцев/регистраций.
7. История объявлений с фото, ценами, пробегами.
8. Коммерческое использование: такси, каршеринг, лизинг.
9. ТО, страхование, диагностика, если есть.
10. Расхождения и качество данных.
11. История обновлений отчета.

Критичные пустые разделы показываются как "данных не найдено на дату обновления". Второстепенные пустые разделы скрываются.

Если разные загруженные отчеты противоречат друг другу, конфликт не скрывается. Отчет показывает:

- какие значения отличаются;
- даты фиксации;
- какой вариант свежее;
- где данных недостаточно.

### 10.2 Оценка прозрачности истории

Оценка сверху - это оценка прозрачности истории, а не объективная оценка технического состояния автомобиля.

Формат:

> 3.8 / 5 · Есть вопросы

Оценка ставится только если данных достаточно:

- есть VIN;
- есть свежий отчет не старше 90 дней;
- распарсены минимум 3 ключевых блока данных.

Ключевые блоки:

- характеристики;
- объявления/пробеги;
- юридические проверки/ДТП/риски.

Если данных недостаточно, вместо числа показывается:

> Недостаточно данных для оценки.

Фактические найденные блоки при этом все равно отображаются.

Факторы оценки:

- свежесть последнего отчета;
- количество подтверждающих загрузок;
- противоречия в пробегах, объявлениях, характеристиках, VIN/госномерах;
- рисковые события: ДТП, расчеты ремонта, ограничения, залоги, розыск, такси, каршеринг, лизинг;
- история объявлений: частые перепродажи, сильные скачки цены/пробега, сомнительные описания;
- полнота данных: фото, объявления, пробеги, владельцы, юридические проверки.

## 11. Данные и хранение

Данные хранятся гибридно:

- ключевые блоки строго структурированы;
- редкие и нестандартные поля хранятся в нормализованном JSON.

Структурированные блоки с первого дня:

- паспорт автомобиля;
- идентификаторы VIN и госномера;
- загрузки отчетов;
- объявления и фотографии;
- пробеги;
- ДТП и расчеты ремонта;
- ограничения, розыск, залог;
- владельцы и регистрационные действия;
- коммерческое использование;
- платежи, подписки, баллы;
- доступы, share-ссылки, жалобы.

Все версии фактов хранятся как наблюдения/события. Актуальный пользовательский отчет строится как сводка поверх этих наблюдений.

### 11.1 Фотографии

Фотографии являются важной частью пользовательской ценности и должны попадать в сводный отчет.

Если в загруженном отчете нет фотографий или их мало, а пользователь пришел по ссылке объявления, фотографии скачиваются из объявления и добавляются в историю автомобиля.

Правила:

- фотографии в нормализованном отчете хранятся бессрочно, если нет жалобы или правового основания удалить;
- дубли фото не хранятся;
- используется hash/perceptual hash;
- одно фото может быть связано с несколькими событиями;
- по жалобе скрывается или удаляется спорное фото, а факты объявления остаются, если с ними нет отдельной проблемы.

## 12. Персональные данные и приватность

Персональные данные людей не попадают в пользовательские отчеты.

Не показывать и по возможности не хранить в нормализованной публичной истории:

- телефоны продавцов;
- имена продавцов;
- аккаунты продавцов;
- адреса;
- переписки;
- контакты;
- другие сведения, относящиеся к людям.

VIN и госномер считаются идентификаторами автомобиля. В полном отчете их можно показывать полностью. В превью VIN показывается частично скрытым, госномер не показывается.

Полные отчеты, VIN-страницы, результаты поиска и share-ссылки не индексируются поисковиками.

## 13. Админка

Админка MVP desktop-first.

На старте используется один админ-пользователь. Действия админа обязательно логируются. Роли можно заложить в базе на будущее, но UI и права в MVP не усложняются.

### 13.1 Очереди

Очереди админки:

- загрузки, которые не распарсились;
- отчеты с подозрением на подделку или изменение;
- отчеты без даты формирования или с подозрительной датой;
- частично распарсенные отчеты;
- конфликты VIN, госномера или характеристик;
- спорные связи объявление -> VIN;
- массовые дубли одного report_fingerprint;
- жалобы пользователей;
- обращения поддержки;
- спорные фото или персональные данные;
- ручная корректировка начислений баллов.

### 13.2 Действия

Действия админки:

- подтвердить или отклонить загрузку;
- начислить, не начислить или отозвать балл с причиной;
- подтвердить или отклонить связь объявления с VIN;
- объединить или разделить ошибочно склеенные данные;
- скрыть фото, событие или спорный блок;
- отправить на повторный парсинг;
- пересчитать сводный отчет и оценку;
- ответить на тикет через Telegram или Max;
- посмотреть оригинал отчета в пределах 6 месяцев;
- создать административное исправление факта с причиной.

Ручная правка фактов - крайний случай, когда автоматическая проверка не справляется. Исходные данные не переписываются тихо. Исправление сохраняется отдельной административной записью с автором, датой, причиной и аудитом.

## 14. Жалобы и поддержка

Кнопка "Пожаловаться" есть в полном отчете и share-версии.

Причины жалобы:

- неверные данные;
- персональные данные;
- фото или контент нарушает права;
- другой автомобиль;
- техническая проблема;
- другое.

Правила обработки:

- жалобы на персональные данные или права на фото временно скрывают спорный контент сразу;
- жалобы на неверные факты оставляют данные видимыми до проверки;
- все жалобы уходят в очередь админки.

Поддержка:

- пользователь может обратиться через форму или Telegram/Max бота;
- обращение попадает в админку;
- ответ отправляется пользователю через выбранный канал.

## 15. Аудит и логи

Бизнес-аудит хранится долго по юридической необходимости:

- платежи;
- подписки;
- начисления и списания баллов;
- доступы к отчетам;
- share-ссылки;
- админ-действия;
- жалобы;
- юридически значимые события.

Технические логи хранятся максимум 30 дней.

В технических логах не должно быть лишних персональных данных и содержимого отчетов.

## 16. SEO и публичные страницы

SEO нужен в MVP, но без блога.

Публичные страницы, написанные вручную в коде:

- главная;
- проверить авто по VIN;
- проверить авто по госномеру;
- проверить авто по ссылке объявления;
- тарифы;
- FAQ;
- документы;
- контакты.

SEO-индексация разрешена только для публичных страниц и безопасного контента.

Не индексировать:

- отчеты;
- VIN-страницы;
- результаты поиска;
- share-ссылки;
- кабинет;
- админку.

Нужны sitemap.xml и robots.txt.

### 16.1 Футер и документы

В футере должны быть:

- пользовательское соглашение;
- оферта;
- политика обработки персональных данных;
- согласие на обработку персональных данных;
- правила сервиса и баллов;
- политика жалоб и удаления;
- контакты.

Документы финально готовит или проверяет юрист.

## 17. Техническая архитектура

Стек:

- backend: TypeScript + Bun;
- frontend: React + Vite;
- админка: React + Vite;
- база: PostgreSQL;
- ORM: Drizzle;
- очереди и фоновые задачи: Redis/queue;
- файлы: S3-compatible storage;
- защита: Yandex SmartCaptcha при подозрительном поведении.

Backend - простой монолитный HTTP API в стиле Hono/Elysia, без тяжелой enterprise-структуры.

Frontend пользовательской части - mobile-first.

Админка - desktop-first.

Приложение stateless. Файлы не хранятся на Railway. Ранняя разработка и закрытый MVP могут жить на Railway, но архитектура не должна быть привязана к Railway.

### 17.1 Backend-модули

- auth: Telegram, Max, телефон, гостевые сессии, привязка каналов;
- search: ссылка, VIN, госномер, кандидаты, snapshot объявления;
- uploads: PDF/authless-ссылки отчетов, проверки, статусы;
- parsing: парсеры Автотеки, затем Авто.ру и Дром;
- vehicles: VIN, паспорт авто, сводный отчет, объединение фактов;
- points: начисления, списания, антидубли;
- billing: тарифы, подписки, платежный провайдер;
- access: открытые отчеты по VIN, share-ссылки, PDF export;
- notifications: Telegram, Max, тикеты, уведомления;
- admin: очереди, модерация, аудит;
- seo/legal: публичные страницы, документы, sitemap, robots.

### 17.2 Инфраструктура и безопасность

Нужно заложить:

- hidden rate limits на поиск, получение доступа к новым отчетам, загрузки и share;
- детект перебора VIN и госномеров;
- анализ подозрительных устройств/IP;
- Yandex SmartCaptcha только при подозрительном поведении;
- noindex для приватных страниц;
- ежедневные бэкапы PostgreSQL;
- проверку восстановления бэкапов;
- S3 backup/versioning для критичных файлов;
- хранение секретов отдельно от кода.

## 18. Аналитика

Минимальная продуктовая аналитика:

- поиски по типу: ссылка, VIN, госномер;
- найдено/не найдено;
- количество кандидатов;
- загрузки успешные/ошибочные;
- причины ошибок парсинга;
- начисления баллов;
- списания баллов;
- получение доступа к новым отчетам;
- повторные просмотры;
- конверсия в регистрацию;
- конверсия в подписку;
- причины жалоб;
- ошибки парсеров;
- антифрод-сигналы.

Персонально чувствительные данные в аналитику не отправляются.

## 19. Открытые юридические проверки перед запуском

Перед публичным запуском нужно отдельно проверить:

- использование фактов из отчетов;
- охрану баз данных;
- использование фотографий из объявлений;
- хранение оригинальных PDF/HTML до 6 месяцев;
- нейтральное упоминание брендов Автотека, Авито, Авто.ру, Дром;
- формулировки SEO-страниц;
- правила начисления баллов;
- оферту и подписки с автопродлением;
- обработку персональных данных;
- жалобы и удаление спорного контента.

## 20. Критерии готовности MVP

MVP можно считать готовым к закрытому запуску, когда:

- работает поиск по ссылке объявления, VIN и госномеру;
- работает минимум один надежный парсер Авито/Автотеки;
- PDF и authless-ссылки проходят базовую проверку;
- начисление баллов работает по утвержденным правилам;
- пользователь может открыть новый отчет за подписочный лимит или балл;
- открытый VIN доступен навсегда;
- сводный отчет отображает ключевые разделы;
- история объявлений с фото работает;
- share-ссылка на 7 дней работает;
- PDF-выгрузка работает;
- Telegram и Max вход/уведомления работают;
- админка покрывает очереди проверки, жалоб и поддержки;
- платежная интеграция работает в выбранном провайдере;
- SEO-страницы, документы, sitemap и robots готовы;
- бэкапы и базовая безопасность настроены.
