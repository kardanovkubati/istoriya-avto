# AGENTS.md

## Язык и коммуникация

- Всегда отвечать на русском.
- Планы, спеки, roadmap-обновления и проектные документы писать на русском, если пользователь явно не попросил другой язык.
- Перед серьезными продуктовыми, frontend, backend или архитектурными изменениями сначала читать roadmap/spec и предлагать план.
- Если задача затрагивает продуктовый сценарий, UX, визуальный дизайн или дизайн-систему, сначала задавать уточняющие вопросы и фиксировать выбранное направление, а уже потом писать implementation plan.
- Не оптимизировать серьезную работу под быстрый demo-эффект.
- Если для качественного результата нужны реальные материалы, которых нет в репозитории или текущей сессии, не упрощать задачу ради быстрого результата. Нужно явно попросить пользователя предоставить нужные картинки, отчеты, файлы, доступы, ключи, права, примеры данных или референсы и объяснить, зачем они нужны.
- Если без недостающих материалов можно сделать только временный или слабый вариант, прямо называть его временным, а не выдавать за готовое продуктовое решение.
- Если задача неоднозначна, сначала классифицировать режим работы:
  - `быстрый фикс`;
  - `серьезная продуктовая фича`;
  - `архитектура / дизайн-система`;
  - `рефакторинг`;
  - `исследование без изменений`.

## Режимы работы

### Быстрый фикс

Использовать для узких багов, текстовых правок, небольших config/test правок.

- Делать минимальное безопасное изменение.
- Не менять архитектуру.
- Не добавлять новые framework/dependency без обсуждения.
- Запускать targeted tests/typecheck по затронутой области.

### Серьезный продуктовый режим

Использовать для новых пользовательских сценариев, экранов, auth/billing/access/report/admin/SEO/legal, схемы данных и интеграций.

- Не начинать с кода.
- Сначала читать roadmap, specs, существующую реализацию.
- Писать или обновлять implementation plan.
- Явно фиксировать продуктовые ограничения и open questions.
- Делить работу на маленькие тестируемые задачи и небольшие коммиты.
- Поведение реализовывать tests-first.
- Перед фразой “готово” запускать обязательные проверки и указывать команды.

### Архитектура и frontend design system

Использовать для новых frontend экранов и крупных UI-flow.

- При работе без Figma-макета и готового visual direction сначала собирать и обсуждать референсы. Не копировать структуру чужого продукта один в один: переносить стиль повествования, качество визуальной иерархии, принципы цветовых сигналов, визуализации и интерактивности, затем накладывать их на продуктовую логику “Истории Авто”.
- Для отчетов и автомобильных сценариев не строить UI из абстрактных одинаковых карточек. Экран должен помогать человеку, который ищет автомобиль: увидеть машину, понять краткий вывод, заметить риски, сравнить факты, загрузить полезный отчет и принять решение перед осмотром или покупкой.
- На главной странице не показывать урезанный `sample report`, `demo report`, пустой макет или декоративную карточку отчета. Если нужно показать отчет на главной, это должен быть полноценный full report для демонстрационного автомобиля: тот же продуктовый experience, те же компоненты, та же структура, глубина и интерактивность, что у отчетов, за которые пользователь тратит баллы или лимит тарифа. Отличие только в том, что данные относятся к вымышленному автомобилю и не раскрывают настоящий пользовательский отчет.
- Не называть полный отчет демонстрационного автомобиля словами `sample`, `demo`, `пример отчета` как будто это макет или заглушка. Допустимые формулировки: `Демонстрационный автомобиль`, `Отчет по демонстрационному автомобилю`, `Полный отчет демонстрационного автомобиля`.
- Полный web-отчет должен быть богаче PDF-выжимки: кликабельные разделы, логичная навигация, серии фотографий, история объявлений, визуализация пробега, юридические статусы, ДТП/ремонт с подходящей визуализацией, источниковая база и раскрываемые детали. Нельзя заменять это одной статичной карточкой с несколькими строками.
- Если нужны реальные фотографии автомобилей, примеры отчетов, PDF, web-ссылки, наборы статусов или данные для графиков, сначала запросить их у пользователя. Не заменять их случайными stock-заглушками, если задача требует продуктового качества.
- Не делать handmade UI для серьезных продуктовых поверхностей без явного согласования.
- Product frontend должен идти через Tailwind + shadcn/ui, если отдельно не утвержден другой стек.
- Сначала согласовать:
  - user flow;
  - layout structure;
  - component boundaries;
  - design-system primitives;
  - empty/loading/error states;
  - accessibility и responsive behavior.
- `App.tsx` должен быть composition/orchestration layer, а не god component.
- Организация frontend:
  - `components/ui` для shadcn primitives;
  - `features/search`;
  - `features/report`;
  - `features/account`;
  - `lib`;
  - `types` только для shared frontend contracts.
- Компоненты должны иметь одну понятную причину меняться.
- Нельзя смешивать API orchestration, formatting helpers, layout sections и feature UI в одном большом файле.

## Backend standards

- Не менять утвержденную продуктовую логику без обсуждения.
- Держать service/repository boundaries.
- Не раскрывать raw artifacts, parser internals, source brands или персональные данные людей в пользовательских responses.
- Idempotency, access boundaries и billing/points logic покрывать тестами до реализации.

## Git discipline

- Работать в feature branch, не в `master`, если пользователь явно не попросил другое.
- Коммиты должны быть маленькими и логичными.
- Не трогать unrelated/untracked файлы.
- Если в рабочем дереве есть чужие изменения, не перетирать их и явно упоминать в статусе.

## Definition of Done

Для серьезной работы:

- План создан или обновлен.
- Тесты добавлены/обновлены до реализации поведения.
- `bun run test` проходит.
- `bun run typecheck` проходит.
- `bun run --cwd apps/web build` проходит, если менялся frontend.
- `bun run --cwd apps/api db:generate` проходит, если менялась schema.
- Код организован под поддержку, а не только под демонстрацию.
